# TDD & Tidy First 원칙

## TDD 사이클
**Red → Green → Refactor**
1. 실패하는 테스트 작성
2. 테스트 통과하는 최소한의 코드 구현
3. 리팩토링 (테스트 통과 후에만)

## Tidy First
변경 유형을 분리:
- **구조적 변경**: 동작 변경 없이 코드 정리 (리네이밍, 추출, 이동)
- **동작적 변경**: 기능 추가/수정

→ 같은 커밋에 섞지 않음. 구조적 변경 먼저.

## 커밋 규칙
- 모든 테스트 통과 + 경고 해결 시에만 커밋
- 작고 자주 커밋
- 구조적/동작적 변경 구분하여 메시지 작성

## 코드 품질
- 중복 제거, 의도 명확히, 단일 책임, 최소한의 상태
